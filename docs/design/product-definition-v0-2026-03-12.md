# 3amoncall プロダクト定義メモ v0

- Date: 2026-03-12
- Source: UI/UX 改修前のヒアリング整理
- Status: Draft

## 1. これは何のプロダクトか

3amoncall は、専任オンコールを置けない個人開発者や少人数チームのための、AI 補助付き incident response console である。

このプロダクトの目的は root cause の学術的な正しさそのものではなく、`MTTR を最小化すること` にある。

したがって、単に observability data を眺めるツールでも、AI が説明してくれるダッシュボードでもない。  
壊れたシステムを安全に早く戻すために、incident 単位で

- 何が壊れているか
- 今何をすべきか
- なぜその提案なのか
- どこを見れば提案を裏取りできるか
- 復旧したかを何で確認するか

を一体で提供する reliability console である。

## 2. 誰のためのものか

主対象は、次のようなユーザーである。

- 個人開発者
- 少人数チーム
- 障害対応専任の担当者を置けないチーム
- observability 専門家ではないが、自分たちで初動対応しなければならない人たち

このため、raw telemetry の読解力を前提にしてはいけない。

## 3. 最重要価値

incident 発生時の最重要価値は、`安全な初動を1つだけ強く提示すること` である。

提案は参考意見として弱く出すのではなく、runbook 代替に近い第一候補として出す。  
ただしブラックボックスではだめで、ユーザーが `なぜその行動を取るべきか` を理解できなければならない。

そのため、推奨アクションは次の 3 層で構造化されるべきである。

- 方針
- 操作
- 実行手順

また、一般論ではなく、できる限り incident とシステム文脈に寄せて提示されるべきである。

## 4. AI の役割

AI は単なる補助情報ではない。  
最初に取るべき行動を宣言的に提示する役である。

ただし最終判断は人間が行う。  
そのため UI の責務は、AI の答えを表示することではなく、`AI の提案を人間が採用判断できるようにすること` にある。

信頼形成に必要な順序は次である。

1. 不確実性の明示
2. 短く明快な根拠
3. evidence へ降りられること
4. chat で深掘りできること

つまり、`強い提案 + 明示的な限界` が必要である。

## 5. incident 時の成功条件

incident 発生時、最初の 30 秒でユーザーが到達すべき状態は、

`何が壊れていて、まず何を止めるべきかが分かること`

である。

最上部で優先されるのは次の順序である。

1. 何が壊れているか
2. 今すぐ取るべきアクション
3. 影響範囲

ここで最初に見せるべきなのは、ユーザー影響とシステム影響である。  
技術原因は重要だが、その裏にあるものとして自然に辿れる構造で提示されるべきである。

## 6. Evidence Studio の役割

Evidence Studio は raw data 保管場所ではなく、`AI 提案の裏取りをする場` である。

重要なのは、ユーザーが telemetry の種類を自分で選ぶことではなく、

- どこを見ればよいかが UI 自体で分かること
- なぜその evidence が重要かが視覚的に理解できること

である。

したがって、Evidence Studio は `Metrics / Traces / Logs / Platform Logs` のデータ種別起点で始まるべきではない。  
次のような意思決定起点で再構成されるべきである。

- 復旧
- 原因
- 証拠
- コンポーネント関係
- 実データ

実データは最初から全量を見せるのではなく、上位カードに応じて絞り込まれて見えるべきである。

## 7. recovery の考え方

incident 対応で重要なのは、原因理解だけでなく `どうなれば戻ったと判断できるか` を持てることである。  
このため incident workspace では、原因だけでなく復旧の見取り図も同時に必要である。

ただし recovery 確認は raw metrics の自己解釈を要求するべきではない。  
AI が定義した最大 3 個の watch item を中心に行うべきである。

watch item の表現も生の指標名ではなく、次の順で構造化されるべきである。

1. 行動ベース
2. 意味ベース
3. 指標名ベース

## 8. AI chat の役割

AI chat は補助説明欄ではない。  
ユーザーが AI 提案に違和感を持ったときの第一の逃げ道であり、見えている evidence の意味を深掘りするための対話層である。

chat の役割は

- 見る場所を案内すること

ではなく、

- 見えているものがどういう意味かを掘ること
- 提案を現場文脈に引き直すこと
- 提案の妥当性を詰めること

にある。

chat は incident workspace の中で軽く見えていてよく、空欄ではなく、小さな会話パネルとして存在し、AI からの最初の一言がある方が自然である。

## 9. 平常時の画面

平常時の 3amoncall は dense dashboard ではない。  
`ambient で editorial な observability surface` である。

ここでは traces / logs / metrics が美しく流れ、サービスの状態感が伝わることが重要である。  
主目的は予兆監視ではなく、普段から生きた運用面を見せながら、異常が起きた瞬間にそのまま incident response へ移れることにある。

## 10. 平常時と incident 時の関係

incident 時は、別ページへ飛ぶ感覚より、平常時画面の上に `ちゃんと作業できる incident workspace` が dramatic に立ち上がる方が、このプロダクトに合っている。

つまり、3amoncall は

- 平常時は ambient な observability surface
- incident 時は act-first な reliability workspace

という二面性を持つ。

## 11. 一言でいうと

3amoncall は observability tool ではなく、`small team 向けの act-first reliability console` である。
