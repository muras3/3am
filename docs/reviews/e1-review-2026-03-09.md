## Findings

### F-E1-001: LeftRail renders `sev-undefined` when severity is not present at the Incident level [BLOCKER]

**File:** `/Users/murase/project/3amoncall/apps/console/src/components/shell/LeftRail.tsx`, line 34

**Detail:**

```tsx
<span className={`sev sev-${inc.packet.severity ?? "critical"}`}>
```

`IncidentPacket` declares `severity` as `z.string().optional()` (see `incident-packet.ts` line 71). The `Incident` interface in `api/types.ts` does not add a top-level `severity` field. So when the receiver creates an Incident from a seeded span, the `severity` on the packet depends entirely on what the anomaly-detector infers — and the seed script does not embed `severity` in the OTLP span payload; the OTLP-to-packet pipeline may produce `severity: undefined`.

The fallback `?? "critical"` suppresses a runtime crash but does not guarantee a defined CSS class actually exists. If the design only defines `.sev-critical`, `.sev-high`, `.sev-warning` etc., then `sev-undefined` will be silently unstyled. More critically, the E2E test in `navigation.spec.ts` line 17 asserts `.incident-item .sev` contains `"open"` — but `.sev` is rendering `inc.status` not `inc.severity`. This test can pass even if severity rendering is broken.

**What to check:** Does `createIngestRouter` / `anomaly-detector` always write `severity` onto the packet it creates? If not, the badge class is unreliable.

**Recommendation:** Move `severity` to the `Incident` record (outside the packet) so the Receiver sets it from anomaly detection, not the packet field. Or guarantee the anomaly-detector always writes a non-optional value. The LeftRail should read `inc.status` and `inc.severity` from distinct fields — currently it mixes them in one `<span>`.

### F-E1-002: Global-setup PID-based teardown is not reliable across Playwright worker restarts [BLOCKER]

**File:** `/Users/murase/project/3amoncall/apps/console/e2e/global-setup.ts`, lines 36–37
**File:** `/Users/murase/project/3amoncall/apps/console/e2e/global-teardown.ts`, lines 2–8

**Detail:**

`E2E_RECEIVER_PID` is stored in `process.env`, which is only available in the global-setup process. Playwright's `globalTeardown` runs in a separate Node.js worker process spawned by the Playwright test runner. The `process.env` mutation from `globalSetup` does not survive into `globalTeardown`'s process on all versions of Playwright (especially when `globalTeardown` is in a separate module path). This means `pid` in teardown may be `undefined`, the `process.kill` call is silently skipped, and the receiver process is orphaned.

Orphaned receiver processes on port 4319 will cause `EADDRINUSE` on the next `e2e` run, which is a hard failure. The `reuseExistingServer: false` config in `playwright.config.ts` ensures the webServer is started fresh each run, but the receiver is started manually outside Playwright's lifecycle control.

**Recommendation:** Write the PID to a temp file (e.g. `os.tmpdir()/3amontest-receiver.pid`) in globalSetup and read it back in globalTeardown. This survives process boundaries reliably. Additionally, add a `SIGKILL` fallback after a timeout in case the process does not exit cleanly on `SIGTERM`.

### F-E1-003: `TracesView` truncates `spanId` without overflow protection [WARN]

**File:** `/Users/murase/project/3amoncall/apps/console/src/components/evidence/TracesView.tsx`, line 55

```tsx
<div className="ta-span">{t.spanId.slice(0, 12)}...</div>
```

If `spanId` is shorter than 12 characters (e.g. `"seed_span_01"` is exactly 12 chars), the ellipsis is still appended: `"seed_span_01..."`. This is cosmetically wrong for the seeded fixtures. The seed script uses spanIds like `"seed_span_01"` (12 chars) — the result is `"seed_span_01..."` which looks truncated when it is not.

**Recommendation:** Guard with `t.spanId.length > 12 ? t.spanId.slice(0, 12) + "..." : t.spanId`.

### F-E1-004: `PlatformLogsView` ignores `platformEvents` on the packet — gate condition partially unmet [WARN]

**File:** `/Users/murase/project/3amoncall/apps/console/src/components/evidence/PlatformLogsView.tsx`

**Detail:**

The E1 gate specifies: "Platform Logs tab: テーブルヘッダー + 'No platform events' 行". The component satisfies this for the static empty-state case. However, `incident.packet.evidence.platformEvents` exists (typed as `z.array(z.unknown())`) and is always `[]` in all 5 fixtures. The component has an `// eslint-disable-next-line @typescript-eslint/no-unused-vars` annotation suppressing the unused `incident` prop, meaning when real `platformEvents` data eventually appears it will silently not render.

This is not a blocker for E1 since the gate only requires the empty state. But the `eslint-disable` comment is a maintenance smell — it indicates the component was designed to ignore its data prop permanently. A comment explaining this is by design (matching the gate spec) would prevent future confusion.

**Recommendation:** Add an inline comment: `// platformEvents ingest is not yet active (Phase E). Static empty state is intentional per E1 gate.`

### F-E1-005: Seed script scenario 05 (CDN cache poison) triggers anomaly detection via `spanStatusCode: 2` alone, not via HTTP status [WARN]

**File:** `/Users/murase/project/3amoncall/apps/receiver/src/scripts/seed-dev.ts`, lines 103–117

**Detail:**

Scenario 05 seeds a span with `durationMs: 50` and `httpStatusCode: 503`. The anomaly-detector triggers on `spanStatusCode: 2`. This is correct and will create an incident. However, the scenario's narrative centers on the CDN serving stale 503s — the distinguishing signal is that the CDN span is fast (50ms) while the origin returns 200. The seeded span cannot capture this two-span relationship because the seed sends only one span per scenario.

The resulting packet will have `primaryService: "cdn-edge"` but only one `representativeTrace` (the CDN span). The fixture packet in `05-cdn-cache-poison.ts` documents two traces (CDN + origin) which are not actually seeded. This creates a mismatch: the E2E test operates on the live-generated packet (one trace) while the fixture documentation shows two traces. The `TracesView` waterfall will show one row for scenario 05, not two.

This is not a correctness failure for E1 (the gate only requires "real data waterfall" rendering), but it weakens the fidelity of scenario 05 as a demonstration fixture.

**Recommendation:** Document this known limitation in a comment in `seed-dev.ts` near the scenario 05 entry: `// Note: CDN+origin two-span narrative requires multi-span ingest not yet implemented. Seeds the CDN span only.`

### F-E1-006: E2E `evidence-studio.spec.ts` Logs tab test depends on first incident having at least one `triggerSignal` [WARN]

**File:** `/Users/murase/project/3amoncall/apps/console/e2e/evidence-studio.spec.ts`, line 43

```ts
await expect(page.locator(".log-row").first()).toBeVisible();
```

This test navigates to the first incident (whichever the receiver returns first from `GET /api/incidents`) and asserts a `.log-row` is visible. The seed order is serial, so the first incident created will be scenario 01 (rate-limit-cascade), which has two `triggerSignals`. However, if seed ordering changes or the receiver returns incidents in a different order (e.g. reverse insertion), a scenario with `triggerSignals: []` would produce an `EmptyView`, and this test would fail with a confusing locator miss rather than a clear assertion.

All 5 current fixtures have `triggerSignals.length >= 2`, so this passes today. But the test is brittle — it asserts on "first incident" without asserting which incident it is.

**Recommendation:** Either (a) assert on a specific incident ID by navigating directly, or (b) add a `data-testid="log-row"` or a count assertion to make the dependency explicit. At minimum, add a comment: `// Relies on scenario 01 (rate-limit-cascade) being the first incident; it has 2 triggerSignals.`

## Recommendation

**Do not mark E1 complete.** Address the two blockers first:

1. **F-E1-001** (severity badge): Audit the anomaly-detector's packet creation path to confirm `severity` is always written as a non-optional string. If it can be `undefined`, either (a) add `severity` as a required field on the `Incident` record independent of the packet, or (b) make the Receiver's incident-formation step always assign a severity. The LeftRail badge CSS class must be deterministic.

2. **F-E1-002** (PID teardown): Replace `process.env["E2E_RECEIVER_PID"]` with a temp file write/read pattern. This is a 10-line change and prevents CI flakiness from orphaned receiver processes.

Warnings F-E1-003 through F-E1-006 can be addressed in the same PR or deferred to the next phase without blocking E1, but F-E1-003 (ellipsis guard) is a trivial fix worth including.

Once the two blockers are resolved, E1 can be approved. The overall implementation quality is good: the data flow from `IncidentPacket` through the API types to the UI components is correct, the fixture variety is appropriate for the 5 validation scenarios, and the E2E spec coverage is meaningful rather than superficial.
