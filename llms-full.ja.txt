# 3am — AI エージェント向け 完全セットアップガイド

OTel データ + LLM を使って、サーバーレスアプリのインシデントを 5 分以内に診断します。

## 3am とは

3am はサーバーレスアプリケーション向けのオープンソースインシデント診断ツールです。以下を行います:

1. OpenTelemetry (OTel) のスパン、ログ、メトリクスを OTLP/HTTP 経由で収集する
2. 異常を検出し、インシデントパケットを自動形成する
3. LLM ベースの根本原因分析を実行する（複数プロバイダをサポート）
4. 結果を AI コパイロット付きのリアルタイムコンソールで表示する

## インストールと実行（ローカル）

前提条件: Docker が起動していること、Node.js 20+

```bash
# Step 1: ユーザーのアプリに OTel SDK をセットアップ（対話式 — フレームワークを検出、依存をインストール）
npx 3am init

# Step 2: ローカル Receiver を起動（Docker イメージを pull して実行）
npx 3am local

# Step 3: （任意）デモインシデントを実行して動作確認
npx 3am local demo

# Step 4: コンソールを開く
open http://localhost:3333
```

### `3am init` が行うこと

- ランタイムを検出: Node.js、Next.js/Vercel、または Cloudflare Workers
- Node.js/Vercel: OTel 依存をインストールし、`instrumentation.ts` を作成、`OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:3333` を `.env` に書き込む
- Cloudflare Workers: `wrangler.toml` または `wrangler.jsonc` を更新して Workers Observability を有効化
- 診断モード（`automatic` または `manual`）と LLM プロバイダをプロンプトで確認

### `3am local` が行うこと

- 3am Receiver の Docker イメージを pull して実行する
- Receiver は OTLP 受信とコンソール UI 配信のためにポート 3333 を待ち受ける

### `3am local demo` が行うこと

- 合成のダウンストリームタイムアウトシナリオを注入する
- 実際の LLM 診断を実行する（Anthropic で約 ¥10/回）
- `service.name=3am-demo` を使用（実アプリのテレメトリと混ざらない）

## LLM プロバイダの設定

3am はプラガブルな LLM プロバイダをサポートします。`--provider` フラグがない場合、以下の順で自動検出されます:

| 優先度 | 条件 | プロバイダ | API キーの要否 |
|----------|-----------|----------|-----------------|
| 1 | `ANTHROPIC_API_KEY` が環境変数にある | anthropic | 必要 |
| 2 | `claude` CLI が PATH にある | claude-code | 不要（サブスクリプション利用） |
| 3 | `codex` CLI が PATH にある | codex | 不要（サブスクリプション利用） |
| 4 | `OPENAI_API_KEY` が環境変数にある | openai | 必要 |
| 5 | Ollama が localhost:11434 で稼働 | ollama | 不要（無料・ローカル） |

### どのモードを使うべきか

| 状況 | モード | プロバイダフラグ |
|-----------|------|--------------|
| `ANTHROPIC_API_KEY` を持っている | `auto` | `--provider anthropic` |
| `OPENAI_API_KEY` を持っている | `auto` | `--provider openai` |
| Claude Code サブスクリプションを使う（API キーなし） | `manual` | `--provider claude-code` |
| Codex サブスクリプションを使う（API キーなし） | `manual` | `--provider codex` |
| Ollama をローカル実行（API キーなし） | `manual` | `--provider ollama` |

**重要:** `--mode manual --provider anthropic` は矛盾です。manual モードはサーバーサイドの API キーを**持っていない**ときのためのものです。`ANTHROPIC_API_KEY` があるなら `--mode auto --provider anthropic` を使ってください。

### automatic モードのセットアップ（API キー経路 — 本番で最も一般的）

```bash
npx 3am init --mode auto --provider anthropic
export ANTHROPIC_API_KEY=sk-ant-...
npx 3am deploy vercel
```

### 診断モード

- **automatic** (alias: `auto`): インシデント検出時に Receiver がサーバーサイドで診断を実行します。サーバーサイドの API キー（`ANTHROPIC_API_KEY` または `OPENAI_API_KEY`）が必要です。
- **manual**: 診断は Console UI または CLI からトリガーされます。ローカルブリッジ（`npx 3am bridge`）経由で Claude Code、Codex、Ollama など、サーバーサイドの API キーを必要としない任意のプロバイダをサポートします。

### manual モードのワークフロー（ローカル Receiver）

1. init で manual モードを設定:
   ```bash
   npx 3am init --mode manual --provider claude-code
   ```
2. ローカルブリッジを起動:
   ```bash
   npx 3am bridge
   ```
3. サーバーサイドの API キーが manual 実行より優先されないように Receiver を起動する。
   - ローカル dev では `npx 3am local` が既に `ALLOW_INSECURE_DEV_MODE=true` を設定しています。
   - Receiver を別途起動する場合、トークンなしで Console にアクセスしたいときは自分で `ALLOW_INSECURE_DEV_MODE=true` を設定してください。
   - プロバイダ解決をブリッジ側の manual ワークフローだけで行いたい場合は、Receiver プロセスから `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` を外してください。
4. Console を dev で別途起動する場合、Receiver を指すように設定:
   ```bash
   VITE_RECEIVER_BASE_URL=http://localhost:3333 pnpm --filter @3am/console dev
   ```
5. Console から再実行やチャットをトリガーするか、CLI から手動診断を直接実行します。

### リモート manual モード（デプロイ済み Receiver へのブリッジ）

Receiver が Vercel または Cloudflare にデプロイされているが、診断は Claude Code や Codex のサブスクリプションでローカルに実行したい場合に使います:

```bash
npx 3am bridge --receiver-url https://your-3am-receiver.vercel.app
```

- ブリッジは WebSocket 経由でデプロイ済み Receiver に接続します
- Cloudflare Workers では: 接続維持に Durable Objects を使用
- Vercel では: Node.js の HTTP server upgrade 経路を使用
- 認証トークンは `npx 3am deploy` が保存した資格情報から自動検出されます。明示的に上書きするには `--receiver-url` を渡してください

### CLI からの手動診断

```bash
npx 3am diagnose \
  --incident-id inc_000001 \
  --receiver-url http://localhost:3333 \
  --provider claude-code
```

## 本番デプロイ

### Vercel

```bash
npx 3am deploy vercel
# 非対話 (CI / AI エージェント向け):
npx 3am deploy vercel --yes --no-interactive --json
```

- Neon Postgres が Vercel Integration 経由で自動プロビジョニングされます
- デプロイ後、CLI が Console 用の短命ワンタイムサインイン URL を出力します
- automatic モード用に Vercel 環境変数として `ANTHROPIC_API_KEY`（または別プロバイダの資格情報）を設定してください

### Cloudflare Workers

```bash
# 以下の権限を持つ Cloudflare API トークンが必要:
#   Account Settings:Read, Workers Scripts:Edit, D1:Edit,
#   Cloudflare Queues:Edit, Workers Observability:Edit
export CLOUDFLARE_API_TOKEN=your-cloudflare-api-token
npx 3am deploy cloudflare --yes
```

実行内容:
1. 3am receiver Worker をデプロイ
2. traces と logs 用の OTLP destinations を作成/更新
3. ユーザーの `wrangler.toml` または `wrangler.jsonc` を更新
4. ユーザーの Worker をデプロイし、テレメトリが流れ始める

## 環境変数

| 変数 | 要否 | デフォルト | 説明 |
|----------|----------|---------|-------------|
| `ANTHROPIC_API_KEY` | Anthropic の automatic モードで必要 | — | Anthropic API キー |
| `OPENAI_API_KEY` | OpenAI の automatic モードで必要 | — | OpenAI API キー |
| `RECEIVER_AUTH_TOKEN` | 本番 | 公開前に必ず設定 | Receiver 認証用の Bearer トークン |
| `RETENTION_HOURS` | 不要 | `48` | テレメトリとクローズ済みインシデントの保持時間 |
| `CLOUDFLARE_API_TOKEN` | CF Workers デプロイ時 | — | Cloudflare API トークン |

## アプリの接続

`3am init` の後、計装をロードした状態でアプリを起動します:

```bash
# Node.js
node --require ./instrumentation.js your-app.js

# Next.js / Vercel — instrumentation.ts は Next.js が自動ロード
```

ログには、`@opentelemetry/auto-instrumentations-node` を通した構造化ロガー（pino、winston、bunyan）が必要です。`console.log` はキャプチャされません。

## モノレポ構造

```
apps/
  receiver/     # Hono — OTLP 受信、異常検知、パケタイザ、console API
  console/      # React + Vite SPA — incident board、evidence explorer、AI copilot
packages/
  core/         # @3am/core — incident packet の Zod スキーマ、formation 型
  diagnosis/    # @3am/diagnosis — LLM 診断エンジン
  cli/          # @3am/cli — CLI コマンド (init, local, deploy, diagnose, bridge, demo)
```

## CLI リファレンス

```bash
# 計装セットアップ
npx 3am init                                          # 対話式セットアップ
npx 3am init --mode auto --provider anthropic         # auto モード (API キー)
npx 3am init --mode manual --provider claude-code     # manual モード (サブスクリプション)
npx 3am init --api-key <key> --no-interactive         # CI / スクリプト用セットアップ

# ローカル開発
npx 3am local                                         # ローカル receiver を :3333 で起動
npx 3am local demo                                    # デモインシデント注入 + 診断実行
npx 3am local demo --yes                              # 料金プロンプトをスキップ

# デプロイ
npx 3am deploy vercel                                 # receiver を Vercel にデプロイ（対話式）
npx 3am deploy vercel --yes --no-interactive --json   # CI デプロイ
npx 3am deploy cloudflare --yes                       # Cloudflare Workers にデプロイ
npx 3am-cli auth-link [receiver-url]                  # 新しいワンタイムサインイン URL を発行

# ブリッジ (manual モード)
npx 3am bridge                                        # ローカルブリッジを :4269 で起動
npx 3am bridge --receiver-url <url>                   # リモート receiver への WebSocket ブリッジ
npx 3am bridge --port 4270                            # カスタムポート

# 手動診断
npx 3am diagnose --incident-id inc_000001 --receiver-url http://localhost:3333 --provider claude-code
```

### フラグ一覧

| コマンド | 主なフラグ |
|---------|-----------|
| `init` | `--mode auto\|manual`、`--provider anthropic\|openai\|claude-code\|codex\|ollama`、`--api-key`、`--model`、`--lang en\|ja`、`--no-interactive` |
| `local` | `--port` (デフォルト 3333)、`--yes`、`--no-interactive`、`--receiver-url` |
| `deploy` | `vercel\|cloudflare`、`--yes`、`--no-interactive`、`--json`、`--project-name`、`--auth-token` |
| `auth-link` | `[receiver-url]`、`--json` |
| `bridge` | `--port` (デフォルト 4269)、`--receiver-url`（リモート WS 接続先。資格情報から自動検出） |
| `diagnose` | `--incident-id`、`--receiver-url`、`--provider`、`--model` |

## 通知セットアップ

```bash
npx 3am integrations notifications
```

3am は OSS でセルフホスト型です。推奨される統合モデル:

1. ユーザー自身が Slack app および/または Discord bot を作成する
2. ユーザーがスレッド配信のための最小権限を付与する
3. ユーザーが `npx 3am integrations notifications` を実行する
4. 3am が資格情報を保存し、以降の通知を自動化する

### Slack（OSS ベストプラクティス）

ベンダー管理の共有アプリではなく、ユーザー所有の Slack app を使用してください。

必要なスコープ:
- `chat:write`
- `channels:read`
- プライベートチャンネルを選択可能にする場合は `groups:read`

推奨フロー:

```bash
npx 3am integrations notifications \
  --provider slack \
  --slack-bot-token xoxb-... \
  --slack-channel-id C...
```

セットアップ後に 3am が行うこと:
- 親インシデント通知を送信する
- 親の `ts` を保存する
- `thread_ts` を使って診断の続報を同じ Slack スレッドに投稿する

### Discord（OSS ベストプラクティス）

スレッド配信が必要な場合はユーザー所有の Discord bot を使用してください。

必要な権限:
- `View Channels`
- `Send Messages`
- `Create Public Threads`
- `Send Messages in Threads`
- `Read Message History`

推奨フロー:

```bash
npx 3am integrations notifications \
  --provider discord \
  --discord-bot-token ... \
  --discord-channel-id ...
```

セットアップ後に 3am が行うこと:
- 親インシデント通知を送信する
- その親メッセージから Discord スレッドを作成する
- 診断の続報をそのスレッド内に投稿する

Discord では Webhook モードも依然として存在しますが、「1 インシデント 1 スレッド」という要件がある場合は bot モードが正しい経路です。
