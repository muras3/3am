## 3. 技術理解度マップ

### よく理解している領域

- **ソフトウェアアーキテクチャ設計**: ADRを主体的に管理し「storageDriverはapps/receiver内に収める（独立パッケージにしない）」「mainはリリース専用、developが統合ブランチ」など境界を自分で決定している。「いちいち決定事項にさせないでまずあなたは網羅的な検討をできてるかどうかを疑いなさい」と命令形で設計姿勢を指示する場面が多数（Claude Code, 3amoncall）
- **CI/CD・GitOps**: 「ブランチを切ってPRを出してください。Vercel側で正常にデプロイできるところまで検証することも計画に入れてください」「CIが失敗してない？ここ成功するまで終わりじゃねーぞ」と具体的に守備範囲を指示（Claude Code, probe/proto_texteditor）
- **LLM評価・ベンチマーク設計**: 「一つのデータソースからでは判別できないのを大前提として、より難しい障害を考えてみてください」「赤ニシンがあること伝えたらそりゃ精度上がるのでは」とLLM診断の落とし穴を自分で気づいて指摘（Claude Code, probe-investigate）
- **AIエージェント活用の原則**: 「TeamCreateはトークンコストが高い。以下を全て満たす場合のみ使え」とCLAUDE.mdに自分でポリシーを定義し、Claude側に判断軸を渡している。トークン消費の問題を独力で発見し最適化ルールを策定（Claude Code, murase）

### 基本的に理解しているがAI任せの領域

- **OTel（OpenTelemetry）**: 「otelのベストプラクティス的なものがあればそれも参考にしたい。log span metricsの中で問題が起きていることが明示的にわかる標準的なエラーみたいなのってないですか」と概念は把握しているが具体的な仕様はAIに調査を委任（Claude Code, 3amoncall）
- **Cloudflare Workers / Vercel デプロイ**: 「vercelとcloudflareは外せない」と制約は明確だが「private linkみたいな機能はないからインターネット通信必須ってお前が言ってなかったっけ」と確認を繰り返すケースがある（Claude Code, 3amoncall）
- **Drizzle ORM / StorageDriver 抽象化**: 採用を決定はしているが実装詳細はAIに委任。「storagedriverも一つに含めるよね」「storage driverってなんだっけ」と確認が入る（Claude Code, 3amoncall）
- **GAN / 音声生成（HiFi-GAN, Qwen3-TTS）**: 「GAN学習って何」「GANが崩壊ってどういう意味」から始まりながら、1日で比較検証の設計まで進んでいる。基礎はAI経由で習得しながら判断は自分でしている（Claude Code, voice-style-transfer）

### 学習中・理解が曖昧な領域

- **Claude Code のトークン消費構造**: 「3/13 12pmにリセットなのに、もう23%も使ってる」「今週のlimitとsonnetのlimitって何が違うの」「opus limitなんてないだろ」と繰り返し混乱が見られる（Claude Code, 3amoncall）
- **WASM / Rust のWeb活用**: 「WASMって結局商用で利用されてるかなあ」「wasmと合わせて。そう言った未来も含めてその結論でいいのかcodexに聞いて」と技術選定判断をAI・Codexに丸投げしている（Claude Code / Copilot Chat, ddb1dfbb090c）
- **TanStack Start / フレームワーク詳細**: 「tanstack startって何。何が優れているの。何を解決してるの」「よくわからない。フロントエンドはどれ？」と詳細を把握していない（Claude Code, 3amoncall）

## 4. プロンプティング力の評価

### 強み

- **終了条件・完了条件の明示**: 「完了条件は、すべてのrunが正常に完了し、診断結果も正常にまとめられること」「バイナリで全ての出力パターンが当初のデザインドキュメントと動きが一致していることを全て確認するまであなたは終われません」と検証基準を事前に定義するパターンが定着している（Claude Code, probe）
- **役割・権限の明確な委任**: 「codexがOKと言わない限り終わらない」「Codexが自律判断する（ユーザー指示がなくても可）」と判断軸をAIに渡すことができている。自分で決めるべきことと任せるべきことの分離が明確（Claude Code, 3amoncall）
- **制約と非目標の明確化**: 「vercelとcloudflareは外せない」「ossっぽくしたいなあ」「ユーザーの管理コストは極限まで低くしたい。意識もさせたくない」と境界条件を最初に置く（Claude Code, 3amoncall）
- **段階的設計→実装の分離**: 「プランが終わったら、コードを書くのは禁止です」「まず方針を出して、codexにレビューした上での方針を再度出して。実装はダメ」と設計フェーズと実装フェーズを明示的に分離するようになっている（Claude Code, voice-style-transfer / 3amoncall）
- **批判的自己評価の要求**: 「ドライに評価してみて」「フラットに振り返ると、anthropicの知見を生かしたルールになっているか？」「あなたから見て、そのプランはclaude codeの生産性を最大限活かせてると判断していますか？」と逆質問で品質を引き出している（Claude Code, probe/3amoncall）

### 改善ポイント

- **短文・単語だけの指示が多い**: 「a」「1」「2」「C」「Bをやろう」「まあ一旦それで」のように文脈依存度が高い短文を繰り返すケースが多数。前のターンの内容が記憶にある前提で使いがちで、コンテキスト切れ・セッション再開時に混乱が生じている。
→ **改善提案**: 意思決定を伝える短文の場合、「プランB（honoのみ）でいい」のように選択肢のラベルを含める。
- **同じ質問の重複送信**: 「なるほどね。ちなみにこれは昨日始めたプロジェクトで…」が3回連続して送信されているケースや、`/usage` `/usage` `/usage` と連打するケースがある。
→ **改善提案**: レスポンスを待たずに送信するクセを意識し、特に進捗確認系は1回に集約する。
- **要求の後出し・変更**: 「実装はダメ」「いや、そうじゃない」「ごめん中止、何してんの？」と方針変更が途中で入るケースが散見される。特にエージェントが動き出してからの変更は損失が大きい。
→ **改善提案**: エージェント・Codexを起動する前に「今日やること」「やらないこと」を1〜2行で書き出す習慣をつける。
- **進捗確認の頻度が高い**: 「順調？」「まだかい」「まだ？」「何してるの？」「今みんな何してるの」「全体の進捗率はどれくらい？」がmintcheckで短時間に集中している。長時間タスクへの不安から発生していると推測される。
→ **改善提案**: 起動時に「完了したら通知して」「中間報告は不要」と明示する。

### 特徴的な癖

- **「codex（GPT）に相談してから」を挟む習慣**: ほぼ全プロジェクトで重要な決定の前後に「codexに聞いてみて」「codexと合意するまで議論して」を挟む。第三者レビューを習慣化している点は強みだが、全案件に適用することでスループットが下がる場合もある（Claude Code, 全プロジェクト）
- **「並列化」への強い志向**: 「極限まで並列化して」「理論値を目指して」「エージェントの待機時間は理論値を目指して」と並列実行を求めるが、実際のファイル競合や待機ロジックで問題が起きている（probe/3amoncall）
- **日本語と英語の混在**: 技術用語は英語（ADR, StorageDriver, OTel, GAN）、会話は日本語で一貫している。エージェントへの指示文も日本語が基本で、ペーストしたログ・エラーメッセージのみ英語というパターン
- **Claude Codeへの感情的関与**: 「最悪だね」「あんたのミスとしたいな」「すまんな怒って」「不快な気持ちにさせましたか」とネガティブフィードバックの後に謝罪するパターンが複数回見られる

## 5. AI活用スタイル

### 主体的に活用できている領域

- **プロダクト設計・ADR策定**: 3amoncallプロジェクトで「ストレージDriverの境界をどこに置くか」「thin eventの責務」「主サービスの判定ロジック」などをAIと議論しながらも最終判断は自分で行っている。「いちいち決定事項にさせないでまずあなたは網羅的な検討ができてるかどうかを疑いなさい」という発言はオーナーシップを保持している証左（Claude Code）
- **品質ゲートの設計**: 「codexのOKが出ない限り終わらない」「PRにコメントを残す」「セキュリティレビューを入れる」と品質チェックポイントを設計したのはユーザー側。AIが提案したわけではない（Claude Code, probe/3amoncall）
- **評価フレームワークの設計**: probe-investigateで「8pt max / 4軸の採点基準」「赤ニシン（ヒント情報）が入ると精度が上がるのは当然」と評価の公平性を主体的に担保している（Claude Code）

### AIへの依存が高い領域

- **フレームワーク・技術選定の根拠**: 「tanstack startって何？honoに統一できない理由は？」「rustのような低レイテンシとかも実現できるの？wasmでもできるよね」と技術スタックの比較検討をほぼ全面的にAI（Claude + Codex）に依存している。
→ **自立に向けた提案**: 採用技術（Hono, TanStack Router, Drizzle）それぞれについて、公式ドキュメントのQuickstartを一度自分で通す。理由の語れない技術選定はADRに書きにくく後から変更コストが高くなる。
- **エラーメッセージの解読**: 「これ何のエラー？」「これはなぜ起きている？」とエラー全文を貼り付けるパターンが初期から継続している（Copilot Chat期）。Claude Code移行後は減少しているが残存している。
→ **自立に向けた提案**: エラーを貼り付ける前に「何が原因だと思うか仮説を1行書く」習慣をつけると診断力が上がる。
- **トークン使用量・APIプランの仕組み**: `/usage` を頻繁に呼び出し、同じ「週次リセット」「Sonnetリミットとは何か」の質問を繰り返している。
→ **自立に向けた提案**: Anthropic の公式ドキュメント（usage limits）を一度自分で読む。CLAUDE.md に理解した内容をメモしておくと同じ質問を繰り返さずに済む。

### ツール別の活用傾向

- **Claude Code（主力）**: アーキテクチャ設計・実装・レビュー・ADR策定・検証ハーネス構築と、ソフトウェア開発の全フェーズをカバー。エージェントチーム機能（TeamCreate）・Skills・Cron等の先進機能を積極的に試している
- **GitHub Copilot Chat（補助・初期のみ）**: 2025年6月〜8月の proto_texteditor 初期開発でコードエラーの即時解決に使用。Claude Code 移行後はほぼ利用なし。Copilot Chat はターミナルから離れた質問・スクリーンショット添付で使い勝手が良い場合があり、一定の使い分けが成立していたと推測される

## 6. 成長の軌跡と学習提案

### 観察された成長

- **2025年9月（探索期）→ 2026年1月（移行期）**: proto_texteditor で「このCLIの最新バージョンは何」「/model」「sonetの性能が知りたい」とツール自体の使い方を探っている。Copilot Chat で作業してきた開発者がClaude Codeに移行している段階と推測される
- **2026年2月（probe / voice-style-transfer）**: エージェント並列化・Codexレビュー組み込み・設計フェーズと実装フェーズの分離が習慣化し始めた。「実装はCodexのレビュー後じゃないとダメ」という制約を自分でルール化している
- **2026年3月（3amoncall / probe-investigate）**: ADR駆動の開発プロセス、「codexがOKを言わない限り終わらない」という品質ゲート設計、CLAUDE.mdへのポリシー外部化（opus使用判断の委任など）が整備された。「Phase完了を自分では宣言しない」という規律も確立している
- **3月13日（本日）**: このレポートをスキルとして登録・実行する行動が見られる。AIとの対話そのものを分析・改善の対象にするメタレベルの取り組みに移行している

### 繰り返し発生する課題

- **コンテキスト消耗後の状態確認**: セッション再開・コンテキスト圧縮後に「あれ、次のアクションは？」「残課題なんだっけ」「今どうなってるんだろ」が必ずと言っていいほど発生している。セッション境界の情報損失が繰り返しの生産性低下を引き起こしている
- **エージェント起動後の変更・中断**: 「ストップ。追加したい依頼がある」「ごめん中止、何してんの？」「手動で落としました」とエージェントを途中で止めるケースが複数回。起動前の要件確認不足が原因と推測される
- **「並列化したい」と「競合・待機ロジック」のギャップ**: 「極限まで並列化してください」と要求するが実際のファイル競合や「それはプランとしておかしくない？並行でできるっていっておいてファイル競合してんだろ」というフィードバックが繰り返されている

### 推奨される次のステップ

1. **セッション開始時のスコープ宣言を習慣化する**: 毎回のセッション冒頭に「今日やること / やらないこと / 完了条件」を3行以内で宣言してから指示を始める。「今日のスコープを明示してもらう」というCLAUDE.mdのSession Hygieneは存在するが、実際のログでは宣言なしに作業が始まるケースが多い
2. **並列化可能性の事前チェックをルール化する**: エージェントチームを起動する前に「変更対象ファイルが重複していないか」「待機依存関係があるか」を確認するチェックリストをCLAUDE.mdに追加する。現状「並列化して」の指示がそのまま渡っており競合が後から発覚している
3. **技術選定メモの蓄積**: 採用した技術（Hono, TanStack Router, Drizzle ORM）について「なぜ選んだか」を1〜2行で自分の言葉でADRに残す習慣をつける。現状AIが書いたADRをそのままコミットしているケースが多く、後から「なんだっけ」の質問が繰り返されている

## 7. 総合評価サマリー

全期間で3,231件のメッセージを分析した結果、Murase さんは **設計思考の強度と品質ゲート設計力** が際立っていると推測される。ADR駆動の開発、「codexがOKを言わない限り終わらない」という外部レビューの義務化、フェーズ完了を自分で宣言しないという規律は、個人開発者としては珍しい高度なプロセス設計であり、probe-investigateの平均9.2/10という診断精度はその成果と見なせる。

一方、**セッション境界での情報損失** と **短文・後出し変更によるエージェントの無駄稼動** が繰り返しのコストになっている。「順調？」「まだかい？」「中止」という言葉が多いのは、長時間タスクへの不安とコンテキスト喪失への対処であり、セッション開始時のスコープ宣言と明示的な「完了通知以外は報告不要」指示で大幅に改善できると推測される。

技術理解においては、アーキテクチャの境界設計とLLM評価は熟知しているが、使用ツール（Hono/TanStack/Drizzle）の内部実装やAPI料金体系はAI依存が高く、同じ質問の繰り返しが生じている。本人が認識しているとおり「最後の5%の運用導線確認を甘くしない」ことが現時点での最大の成長余地と推測される。
